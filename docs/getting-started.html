Draft summary

Internet Protocol for Alarm Management

OpenIPA is intended as a simple, open, IP-based replacement for ESPA 4.4.4. The published draft focuses on straightforward XML messages carried over a lightweight TCP framing layer, so alarm integrations can move from serial-era assumptions to modern IP infrastructure without becoming overly complex.

This documentation reflects the current draft. It tightens wording and removes obvious contradictions, but it does not claim new protocol behavior where the draft is still incomplete.

Example

Smallest useful message

The simplest published example is a text message with no acknowledgement requested.

<TextMessage To="123456">Text to display</TextMessage>

When transmitted over the TCP framing described in this draft, the payload is preceded by a fixed request line and a `Content-Length` header.

PUT / IPA/1.0
Content-Length: 56

<TextMessage To="123456">Text to display</TextMessage>

The original page also included a C# example. The essential point is that the XML bytes are written after the framing header and empty separator line.

Behavior

Acknowledgement model

If a sender includes an `Id` attribute in a message, it is requesting an acknowledgement. The receiver is expected to return an `Ack` message within `T ACK` seconds and copy the same `Id` value into the acknowledgement.

<TextMessage Id="100" To="123456">Text to display</TextMessage>

A successful acknowledgement uses the `Result` attribute.

<Ack Id="100" Result="200">OK</Ack>

A failure can be returned with another result code, for example:

<Ack Id="100" Result="404">Not found</Ack>

The published draft uses result codes inspired by HTTP status codes, but they are part of the OpenIPA application layer, not HTTP itself.
