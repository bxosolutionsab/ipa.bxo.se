Wire format

Transport overview

OpenIPA sends XML messages over a TCP-based transport that resembles HTTP framing, but the draft treats communication as asynchronous rather than request-response in the strict HTTP sense. The framing exists so the receiver can determine where each XML message starts and ends.

Part Example Description
Request line PUT / IPA/1.0 Fixed line defined by the draft.
Header Content-Length: 52 Mandatory message length metadata.
Separator Empty line Separates headers from the XML payload.
Payload <TextMessage Id="100" To="123">Testing</TextMessage> The XML message itself.

Example

Example session

A sender transmits a message with an `Id`, which requests an acknowledgement.

PUT / IPA/1.0
Content-Length: 52

<TextMessage Id="100" To="123">Testing</TextMessage>

The receiver later answers with an `Ack` carrying the same `Id`.

PUT / IPA/1.0
Content-Length: 35

<Ack Id="100" Result="200">OK</Ack>

Requirements

Published transport requirements

  • The request line is fixed as `PUT / IPA/1.0`.
  • `Content-Length` is mandatory.
  • Other header fields may be added, but peers are not required to understand them.
  • XML is encoded as UTF-8, with pure 7-bit ASCII also allowed by the draft.

Operational notes

Published security notes

The current draft discusses security as three related concerns rather than as a complete mandatory profile.

Identification

If multiple clients connect to the same server, the client may need to identify itself.

<Heartbeat Id="103" ClientId="012345" />

Authentication

The draft sketches authentication as an attribute carried in the message.

<Heartbeat Id="103" Authentication="dz80mi0pql+js9028jm8c89jv8" />

Encryption

The draft explicitly says authentication is of limited value without transport encryption, and recommends using TLS on the TCP socket.
